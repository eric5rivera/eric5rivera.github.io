##Without it, there is anarchy
Clear communication can make or break the efficiency of a team. What’s the point of a team, if the members are not able to collaborate effectively with one another? Having a coding standard makes it clear for people to write code in a format that is readable and compatible.
Readability isn’t just for the person who wrote the code - more importantly, it is for others. There’s some overhead in getting acquainted with the new prose. Exchanging code between members of a team, between teams, or even to strangers incurs overhead. Having a consistent layout of design and comments reduces the overhead.
Benefits in terms of compatibility concern other pieces of code both in the present and future. Ideally, the code we write should be compatible with other parts of a system and also be maintainable for the future. Having a coding standard enables developers to produce code that is compatible, maintainable, and easier to read.

##Here’s the tricky part
What’s a society with rules, without proper enforcement of those rules? What’s a ham sandwich without the ham? What’s a coding standard, without consistent compliance with the coding standard? Merely having a coding standard is not enough to see the benefits. It is in the application of this standard that pays dividends.
ESLint comes to the rescue when it comes to enforcing the coding standard. 

##Behold, the green checkmark!
The green checkmark has been both painful and useful at the same time. It’s the lack of that green checkmark that provides the pain and utility. The line by line critique with the severity of coding standard violations denoted by various colors has been an excellent tool that helped me write cleaner code and save time.
Being new to JavaScript, I have found that having a linter like ESLint has made learning the new language more informative and maybe even more straightforward. I felt as though a person was sitting aside my shoulder to guide me in the process of writing my JavaScript prose. This tool has felt similar to using tools like spell-check in Word, or even Grammarly.

##Conclusion
I believe that coding standards can help someone learning a new programming language. I can support this belief with my anecdotal evidence with ESLint in IntelliJ IDEA. With this new tool on my belt, I have an easier time complying to coding standards. These coding standards provide an array of benefits for my team and me and the present and the future.
